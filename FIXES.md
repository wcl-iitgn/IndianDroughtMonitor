## #5 (primary) — NaN cells were counted as "exceptional drought" (D4)

**Root cause.** `classify_cdi()` used fall-through thresholds (`if v > -0.5 … else return "D4"`).
A masked/`NaN` CDI cell fails every `>` comparison and fell straight through to `D4`. The per-state
tally in `build_summary_context()` then counted those masked cells as drought. On the 2026-06-03
grid, 85.2% of Gujarat's cells were masked (`NaN`), so the summary reported **"Gujarat 85.2% in
drought"** — the published number was literally Gujarat's no-data share, not drought.

(The 2026-05-20 grid had ~1% masked cells, which is why this only surfaced from 05-27 onward when the
masked share jumped to ~8–11%.)

**Fix (`generate_hydro_outputs.py`).**
- `classify_cdi()` now returns `None` for `None`/`NaN` *before* the thresholds, so a no-data cell is
  never classified as any drought category.
- The per-state tally counts masked cells as `nodata` and excludes them; drought percentages are
  computed over **valid cells only**.
- Added a coverage guard: a state is dropped from the summary when it has fewer than 8 valid cells
  **or** valid coverage below `MIN_STATE_COVERAGE` (0.5). Excluded states are reported via a printed
  warning, e.g. `! data-coverage warning: excluded state(s) … Gujarat 85.2%`.

**Verified** on the real 2026-06-03 grid: Gujarat is now excluded (with the warning); the worst-states
list the LLM sees becomes Delhi 69.6%, Arunachal 52.3%, J&K 50.3%, Mizoram 46.6%, Meghalaya 43.5%,
Manipur 42.6%. National figures (driven by the timeseries, not the grid tally) are unchanged. On the
clean 2026-05-20 grid there are **no** false exclusions.

**Defensive fix (`build_districts.py`).** `build_districts.py` had the same fall-through `classify()`.
It was *latent* there (its `load_cdi()` already drops `NaN` rows before they reach the grid), but the
classifier is now hardened the same way (`None` for `None`/`NaN`) and the district tally skips no-data
cells, so the district numbers can't regress the same way if the input path ever changes.

## #1 — `assets/ai/hydro-stats.json` was orphaned / stale (frozen at Feb 2026)

**Root cause.** The chatbot reads `assets/ai/hydro-stats.json`, but nothing in the pipeline ever
wrote it — it was a hand-made file last updated in February.

**Fix (`generate_hydro_outputs.py`).** Added `write_hydro_stats()`, wired into the hydro stage as a new
`stats` step (now in the default `--only` list: `maps,dashboards,summaries,pdf,stats`). It recomputes
the file every hydro run from the **same monthly input grids** the maps/PDF use — the national mean of
each month-column (current / forecast / prev_1..prev_4 / last_year_same_month / driest / wettest),
ignoring masked `NaN` cells — plus a per-parameter label/unit/description and a `generated` date.

**Note on the bundled `hydro-stats.json`.** The copy in this bundle was regenerated by that function
from the real `*_2026_05_31` inputs (month label "May 2026", forecast "June 2026", n_cells 4537). It
replaces the stale Feb file immediately, and your next hydro run will overwrite it from your full
inputs. Cross-check: the values that were `current_month` in the old Feb file now correctly land in
`prev_3` (February is 3 months before May), and the driest/wettest analogues are unchanged.

## #2 — District stats silently lagged the rest of the site

**Root cause.** The `districts` stage is network-dependent and was only run with `--with-districts`
or `--force`, so a routine `build.py` left `data/districts/` frozen. The site was on week 2026-06-03
while district stats were still 2026-05-20.

**Fix (`build.py`).** A default build now checks whether the district data's `week_ending` is older
than the latest week in `data/India_Drought_Area_Timeseries.txt` (or missing entirely) and, if so,
folds `districts` into the build automatically — printing an `[auto] district data is stale …` line.
`--with-districts` / `--force` still force it; `--only …` is unchanged (no auto-add when you've asked
for an explicit stage subset).

## #3 — `data/summary_latest.txt` was missing the regional block

**Root cause.** The hydro stage writes `data/summary_latest.txt` (header + national narrative only).
The regional "Regional conditions this week:" block is computed later, in `generate_summary_pdfs.py`,
and was appended **only** to the archived `data/summaries/summary_<date>.txt`, never mirrored back into
the live `summary_latest.txt`. So the site's latest summary showed the national text but no regional
breakdown (and, before fix #5, the stale national text still naming Gujarat at 85.2%).

**Fix (`generate_summary_pdfs.py`).** The summary stage now identifies the newest week across the whole
index (independent of any `--dates` filter) and, when it writes that week's regional block, mirrors the
full `header + national + regional` text into `data/summary_latest.txt`. `national_part()` strips any
previously-appended block first, so re-runs stay idempotent. Older weeks run with `--dates` will not
clobber `summary_latest.txt`.

(After you re-run the full pipeline, the national text in `summary_latest.txt` is also corrected,
because fix #5 changes the per-state context the LLM is given.)

## #4 — ~900 MB of `_build` intermediates were committed

**Root cause.** The per-language/per-week working dirs `data/summaries/_build/` and
`data/forecast_summaries/_build/` were tracked in git.

**Fix (`.gitignore`).** Added ignore rules for `**/_build/` (and the two explicit paths), plus
`.DS_Store`. These dirs are regenerated each run; the finished PDFs/txt that live alongside them stay
tracked.

To drop the already-committed copies from the repo (keeping them on disk):

    git rm -r --cached data/summaries/_build data/forecast_summaries/_build
    git rm --cached .DS_Store **/.DS_Store 2>/dev/null
    git commit -m "Stop tracking _build intermediates and .DS_Store"

## #6 — coverage / sanity warning

Implemented as part of fix #5: when a state's CDI grid is mostly masked for a given week, the build now
prints a `! data-coverage warning:` line listing the excluded state(s) and their no-data share, so a
bad/incomplete input week is visible at build time instead of silently producing a wrong percentage.
